# Multi-Process Architecture Migration Plan

## 1. Goal
Transition Wisp from a single-process monolithic architecture to a multi-process architecture (Browser Process + N Renderer Processes) to prevent a single tab crash (e.g., in layout or QuickJS) from bringing down the entire browser UI.

## 2. Current Architecture
Currently, Wisp's architecture is tightly coupled:
- `struct browser_window` is the core object representing a window/tab.
- UI events (mouse, keyboard, scroll) are processed in the main thread and directly invoke layout engine functions (`browser_window_mouse_action`, `browser_window_key_press`).
- The Layout Engine (HTML, CSS, JS) directly issues drawing commands to the frontend via the `struct plotter_table` interface.

## 3. Proposed Architecture
- **Browser Process (UI)**: Manages OS windows, tabs, URL bar, extensions. Intercepts OS input events and paints graphics.
- **Renderer Process (Engine)**: Contains the Content Engine (HTML, CSS), Layout Engine, and JS Engine (QuickJS). One process per tab/domain.
- **IPC Mechanism**: Unix Domain Sockets (Linux/macOS) and Named Pipes (Windows) will facilitate communication between the Browser Process and Renderer Processes.

## 4. Required Steps

### Phase 0.5: Interim Threaded Mode (Validation)
- Before implementing full multi-process isolation, validate the display list serialization and IPC logic using an internal task queue.
- This allows for testing the separation of UI and layout without the overhead of `fork/exec` or `CreateProcess`.

### Phase 1.0: IPC Infrastructure & Multi-Process Execution
- Implement a cross-platform IPC library (`src/utils/ipc/`) using Domain Sockets / Named Pipes.
- **Browser Side**: Update window creation to spawn a new Renderer Process (e.g., using `fork()` + `exec()` or `CreateProcess()`) instead of initializing the content engine directly.
- **Renderer Side**: Create a new executable entry point (`wisp-renderer`) that listens on an IPC channel, creates an off-screen `browser_window`, processes navigation requests (`IPC_MSG_NAVIGATE`), runs the layout/JS engine, and streams display lists back to the Browser process.
- **Input Serialization**: Input events generated by the frontend must be serialized and sent to the Renderer over IPC.

### Phase 2.0: Drawing / Display List Serialization & Acceleration
- The `struct plotter_table` (in `include/wisp/plotters.h`) defines drawing commands.
- **Renderer Side**: Implement an "IPC Plotter" (`src/utils/ipc/ipc_plotter.c`) that implements the `plotter_table` interface. Instead of drawing to the screen, it serializes the drawing commands into a display list buffer.
- **Browser Side**: Receive the display list buffer over IPC and replay it onto the frontend's native `plotter_table`.
- **Display List Buffer**: Implement this as a Circular Buffer or a double-buffered shared memory region to manage back-pressure.
- **Command Batching**: Group drawing commands for an entire "Layer" or "Tile" into a single IPC packet to reduce context-switching overhead.
- **Shared Memory (SHM) for Bitmaps**:
  - Renderer decodes an image into a Shared Memory block.
  - Renderer sends an IPC message: `IPC_MSG_BITMAP_READY { shm_handle, width, height }`.
  - Browser maps that handle and renders it directly, synchronized via a Cross-Process Mutex.

## 5. Challenges and Risks
- **Synchronous Returns (Font Metrics)**: The layout engine sometimes expects synchronous returns from the frontend (e.g., text measurements).
  - *Solution*: When the Browser Process starts a Renderer, pass a "Font Table" (character widths for common fonts/sizes) via Shared Memory. The Renderer can then calculate word widths locally without IPC overhead.
- **Latency**: The "Input -> IPC -> Layout -> IPC -> Display List -> UI" loop must stay under 16ms for 60fps scrolling.
- **Process Management ("Zombies")**: The Browser must catch `SIGCHLD` or wait on the process handle if a Renderer crashes. Instead of closing the window, show a "Sad Tab" page and offer a "Reload" button to spawn a fresh renderer.
- **Security**: Once isolated, the Renderer process should be sandboxed (e.g., `seccomp` on Linux, "AppContainer" on Windows) to prevent direct filesystem access.

## Conclusion
This is a massive refactoring effort. The best approach is to start with the "Interim Threaded Mode" (Phase 0.5) to isolate the display list serialization before tackling full multi-process management.
